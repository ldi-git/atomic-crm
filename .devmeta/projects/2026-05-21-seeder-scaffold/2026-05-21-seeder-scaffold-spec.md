# Feature spec — Seeder scaffold (Iteration 1.1)

## Scope

Bootstrap the Supabase seeder plumbing so iteration 1.2 (auth users) and 1.3 (CRM data) can plug into it. After this feature: `make seed` resets the local Supabase DB and runs a stub seeder that verifies the connection is local, verifies the DB is clean, and writes one smoke-test row to `tags`. No CRM data, no auth users — those land in later iterations.

## Architecture

### Files created

- `scripts/seed-supabase.ts` — TypeScript entry point for the seeder. Run via `npx tsx`. Imports `@supabase/supabase-js` directly. Uses `execa` to call `npx supabase status -o json`.

### Files modified

- `package.json` — add `tsx` and `@supabase/supabase-js` to `devDependencies`.
- `package-lock.json` — regenerated by `npm install`.
- `Makefile` — add `seed` target that depends on `supabase-reset-database` and runs the seeder.

### Files NOT touched

Anything under `src/`. Anything under `supabase/`. The fakerest generators (those become inputs to the seeder in iteration 1.3 — read-only here).

### Existing scripts to mirror

`scripts/supabase-remote-init.mjs` already uses `execa` for shelling out to the Supabase CLI. Reuse that pattern in the new TS script.

## Implementation guide

### Step 1: Add dev dependencies

```bash
npm install --save-dev tsx @supabase/supabase-js
```

`@supabase/supabase-js` is currently transitive via `ra-supabase-core@3.5.2` — we want it as a direct dep so the seeder's imports stay stable across upstream changes. Pin to a version compatible with the transitive one (npm should resolve this; if dedup fails, match `2.90.x`).

### Step 2: Write `scripts/seed-supabase.ts`

Outline:

```typescript
import { createClient } from "@supabase/supabase-js";
import { execa } from "execa";

const LOCAL_HOSTS = new Set(["127.0.0.1", "localhost"]);
const SMOKE_TAG_NAME = "__seed_smoke_test__";

async function main() {
  // 1. Resolve Supabase URL.
  //    Prefer process.env.SUPABASE_URL; fall back to parsing from `npx supabase status -o json`.
  const { apiUrl, serviceRoleKey } = await resolveLocalSupabase();

  // 2. Local-URL guard.
  const host = new URL(apiUrl).hostname;
  if (!LOCAL_HOSTS.has(host)) {
    console.error(
      `[seed-supabase] Refusing to run against non-local Supabase URL: ${apiUrl}\n` +
        `              This seeder is local-only by design. Aborting.`,
    );
    process.exit(1);
  }

  const supabase = createClient(apiUrl, serviceRoleKey, {
    auth: { autoRefreshToken: false, persistSession: false },
  });

  // 3. Clean-DB precondition.
  const { count, error } = await supabase
    .from("companies")
    .select("*", { count: "exact", head: true });
  if (error) throw error;
  if ((count ?? 0) > 0) {
    console.error(
      `[seed-supabase] companies table is not empty (${count} rows).\n` +
        `              Run \`make seed\` (which resets first) instead of invoking the seeder directly.`,
    );
    process.exit(1);
  }

  // 4. Smoke-test insert.
  const { error: tagErr } = await supabase
    .from("tags")
    .insert({ name: SMOKE_TAG_NAME, color: "#000000" });
  if (tagErr) throw tagErr;

  console.log("[seed-supabase] OK — smoke-test tag inserted.");
}

async function resolveLocalSupabase(): Promise<{
  apiUrl: string;
  serviceRoleKey: string;
}> {
  // Strategy:
  // - If SUPABASE_URL and SUPABASE_SERVICE_ROLE_KEY are both set, use them
  //   (lets CI / advanced users override).
  // - Otherwise, shell out to `npx supabase status -o json` and parse:
  //     { API_URL: "http://127.0.0.1:54321", SERVICE_ROLE_KEY: "eyJ..." }
  //   (key name may be "service_role_key" or similar — confirm by running it
  //    once before writing the parser).

  const envUrl = process.env.SUPABASE_URL;
  const envKey = process.env.SUPABASE_SERVICE_ROLE_KEY;
  if (envUrl && envKey) return { apiUrl: envUrl, serviceRoleKey: envKey };

  const { stdout } = await execa("npx", [
    "supabase",
    "status",
    "-o",
    "json",
  ]);
  const status = JSON.parse(stdout);
  // Map fields to a stable shape — see open question below.
  const apiUrl: string = status.API_URL ?? status.api_url;
  const serviceRoleKey: string =
    status.SERVICE_ROLE_KEY ?? status.service_role_key;

  if (!apiUrl || !serviceRoleKey) {
    throw new Error(
      "[seed-supabase] Could not extract API_URL / SERVICE_ROLE_KEY from `npx supabase status -o json`. " +
        "Is Supabase running locally? Try `make start-supabase`.",
    );
  }
  return { apiUrl, serviceRoleKey };
}

main().catch((err) => {
  console.error("[seed-supabase] Failed:", err);
  process.exit(1);
});
```

**Notes:**
- `process.exit(1)` is used liberally; this is a CLI script, not a library.
- Auth options on the client are set to no-refresh / no-persist — service-role flow.
- Imports are bare-minimum; no need to depend on the fakerest generators in this iteration.

### Step 3: Confirm the `supabase status -o json` shape

Before writing the parser, run `npx supabase status -o json` once locally and note the exact field names. The Supabase CLI has shipped both `API_URL` and `api_url` shapes over the years. **The task agent must verify the shape and adjust the parser accordingly — do not guess.**

### Step 4: Add `make seed` target

```makefile
seed: supabase-reset-database ## reset DB and load demo seed data
	npx tsx scripts/seed-supabase.ts
```

Insert near `supabase-reset-database` (around line 28). Keep the existing target untouched — both must work.

### Step 5: Verify end-to-end

- `make supabase-reset-database` → empty tables.
- `make seed` → exits 0, one row in `tags`.
- Tampered `SUPABASE_URL=https://example.supabase.co npx tsx scripts/seed-supabase.ts` → exits 1, message includes "non-local".
- `npx tsx scripts/seed-supabase.ts` without a prior reset (run it twice in a row) → second invocation exits 1, message includes "DB not clean".

## Test strategy

No new automated tests this feature — the seeder is its own integration test. Standing tests must remain green:

```bash
make typecheck   # confirms scripts/seed-supabase.ts type-checks
make test        # unit tests untouched
make lint        # eslint covers .ts under scripts/
```

## Open questions

- **Exact key name in `supabase status -o json`.** Resolve at task time by running the command once. Could be `API_URL`, `api_url`, or buried in a `services` array. Don't guess.
- **What to do if `supabase status` fails** (Supabase not running). Current plan: bubble the execa error up with a hint to `make start-supabase`. That's sufficient.
